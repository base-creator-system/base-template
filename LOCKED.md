# BASE SYSTEM — LOCKED RULES
Version: 1.0  
Status: ENFORCED  
Derived from: CANON_PUBLIC.md

This document defines what is **LOCKED** in Base.
If any rule in this file is violated, the build is **not Base**.

---

## 0) Meaning of LOCKED
LOCKED means:
- Must not be altered by pilots
- Must not be overridden by customization
- Must not be bypassed “just this once”
- Changes require Canon update first

No exceptions through convenience, speed, or pressure.

---

## 1) System Identity (LOCKED)
The following identity elements are immutable:
- Base is a creator-owned PWA system
- Base is not a social media platform
- Base does not use algorithmic feeds
- Base does not extract user data for profit
- Base is ownership-first by design

Any feature contradicting this is disallowed.

---

## 2) Repository Authority (LOCKED)
This repository is the system-of-record for Base Core.

LOCKED:
- Canon files define truth, not code comments
- Base Core logic lives above pilots
- Pilots may not redefine Base rules
- No alternate “core” forks may claim to be Base

---

## 3) Core Structure (LOCKED)
The following structural expectations must remain intact:

- Separation of:
  - Core system
  - Framework / UI constraints
  - Pilots / apps
- Pilots live downstream from Core
- Core must not import pilot-specific logic

Breaking structure = breaking Base.

---

## 4) Base UI Behavior (LOCKED)
LOCKED behaviors include:
- Navigation logic and flow patterns
- Base component behavior (cards, buttons, lists, tabs)
- Accessibility baseline (readability, contrast, tap targets)
- Mobile-first behavior expectations

Pilots may style.
Pilots may not redefine behavior.

---

## 5) Customization Boundary (LOCKED)
Pilots may NOT:
- Alter Base navigation logic
- Override Base component behavior
- Remove Base governance references
- Introduce hidden tracking or surveillance
- Inject scripts that violate Base ethics

Customization exists **within** constraints.

---

## 6) Monetization Boundaries (LOCKED)
LOCKED prohibitions:
- Selling user data
- Forced advertisements
- Pay-or-lose-access models
- Manipulative engagement loops
- Core system dependency on monthly payments

Allowed monetization must be optional and documented.

---

## 7) Security & Integrity (LOCKED)
LOCKED requirements:
- No backdoors
- No credential harvesting
- No hidden third-party scripts
- No malware or surveillance tooling

Security violations invalidate the build.

---

## 8) Change Control (LOCKED)
Any change to LOCKED items requires:
1. Written justification
2. Canon update
3. Explicit approval
4. Then implementation
5. Then validation via pilot

Silent changes are forbidden.

---

## 9) Pilot Authority (LOCKED)
Pilots:
- Validate Base
- Do not redefine Base
- Do not weaken Base
- Do not bypass rules

If a pilot conflicts with LOCKED rules, the pilot must change — not the rules.

---

## 10) Final Lock Statement
Base evolves by law, not impulse.

If a build violates this document, it is not Base.
