# BASE SYSTEM — EDITABLE ZONES
Version: 1.0  
Status: ACTIVE  
Counterpart to: LOCKED.md

This document defines what is **EDITABLE** in Base.
Editable zones are where pilots customize freely **without breaking Base**.

If something is not listed here, assume it is governed by LOCKED rules.

---

## 0) Meaning of EDITABLE
EDITABLE means:
- Pilots may customize without approval
- Changes must not alter Base behavior
- Creativity is encouraged within boundaries
- Ownership remains with the creator

Editable is freedom **with structure**.

---

## 1) Visual Identity (EDITABLE)
Pilots may customize:
- App name
- Logos and icons
- Color palettes
- Typography choices (within readability)
- Backgrounds and imagery
- Hero sections and visual themes

Constraints:
- Must remain readable
- Must remain accessible on mobile
- Must not impersonate Base Core identity

---

## 2) Content (EDITABLE)
Pilots fully control:
- Text content
- Links
- Media embeds (audio, video, images)
- Pages and sections built using Base components
- Creator-specific messaging and storytelling

Content ownership belongs to the creator.

---

## 3) Page Composition (EDITABLE)
Pilots may:
- Choose which Base components to use
- Arrange pages and sections
- Create custom pages using allowed components
- Decide information hierarchy and flow

Constraints:
- Base navigation logic must remain intact
- Base components may be composed, not rewritten

---

## 4) Features & Modules (EDITABLE)
Pilots may add:
- Optional tools or integrations
- Creator-specific utilities
- Feature modules that do not modify Base Core
- External links or embeds that respect Base ethics

All additions must:
- Be transparent to users
- Not introduce tracking or surveillance
- Not override Base behavior

---

## 5) Monetization (EDITABLE WITHIN RULES)
Pilots may choose to:
- Offer paid content
- Sell services or products
- Use memberships or subscriptions
- Enable optional paid tools

Constraints:
- Monetization must be optional
- Must not remove access to the creator’s core presence
- Must not violate LOCKED monetization boundaries

---

## 6) Branding & Attribution (EDITABLE)
Pilots may:
- Brand their app fully
- Choose how Base attribution appears (within requirements)
- Present themselves independently

Constraints:
- Base governance references must not be removed
- Base must not be misrepresented as something else

---

## 7) Experimentation (EDITABLE)
Pilots are encouraged to:
- Test layouts
- Explore formats
- Try new presentation styles
- Learn from real use

If an experiment:
- Requires changing LOCKED rules → stop and document first
- Conflicts with Canon → adjust the experiment, not Base

---

## 8) Responsibility of Freedom
With editability comes responsibility:
- Do not harm users
- Do not deceive audiences
- Do not weaken system integrity
- Do not undermine creator ownership principles

Freedom inside Base is protected by law, not chaos.

---

## 9) Final Editable Statement
Creativity thrives where rules are clear.

If a customization stays within this document,
it is valid Base usage.
